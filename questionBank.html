Java is platform independent because of the Java Virtual Machine (JVM). Here's how it works:

Write Once, Run Anywhere (WORA):
- Java source code (.java files) is compiled into bytecode (.class files)
- Bytecode is platform-independent and can run on any system with JVM
- JVM acts as an interpreter between bytecode and the operating system

Key Points:
• JVM is platform-specific (different JVMs for Windows, Linux, Mac)
• Bytecode is the same regardless of platform
• No need to recompile code for different operating systems
• This makes Java highly portable compared to languages like C/C++
JVM (Java Virtual Machine):
• Executes Java bytecode
• Provides runtime environment for Java programs
• Platform-specific component
• Handles memory management and garbage collection

JRE (Java Runtime Environment):
• Contains JVM + core libraries
• Required to run Java applications
• Includes class libraries, loader, and verifier
• Does NOT include development tools

JDK (Java Development Kit):
• Contains JRE + development tools
• Includes compiler (javac), debugger, and other tools
• Required for developing Java applications
• Supersedes JRE for development purposes

Simple Analogy:
• JVM = Engine (runs the car)
• JRE = Car without tools (can drive but can't build/repair)
• JDK = Complete car with tools (can drive AND build/repair)
While Loop:
• Entry-controlled loop
• Condition is checked BEFORE executing the loop body
• May execute zero times if condition is false initially

Do-While Loop:
• Exit-controlled loop
• Condition is checked AFTER executing the loop body
• Executes at least once, even if condition is false

Syntax Comparison:
// While loop
while (condition) {
    // code to execute
}

// Do-while loop
do {
    // code to execute
} while (condition);

Example:
// While loop - may not execute
int i = 5;
while (i < 5) {
    System.out.println("While: " + i);
    i++;
}

// Do-while loop - executes at least once
int j = 5;
do {
    System.out.println("Do-While: " + j);
    j++;
} while (j < 5);
== Operator:
• Compares memory references (addresses)
• Used for primitive types and object references
• Checks if two references point to the same memory location

equals() Method:
• Compares content/values of objects
• Defined in Object class, can be overridden
• Checks if two objects are meaningfully equal

Key Points:
• Always use equals() for object content comparison
• == can give unexpected results for objects
• Override equals() in custom classes for proper comparison

Method overloading is a feature that allows a class to have multiple methods with the same name but different parameters. It's a form of compile-time polymorphism.

Key Characteristics:
• Same method name
• Different parameter lists (type, number, or order)
• Can have different return types
• Must be in the same class

Types of Overloading:
1. Different number of parameters:
void display(int a) { ... }
void display(int a, int b) { ... }

2. Different types of parameters:
void display(int a) { ... }
void display(double a) { ... }

3. Different order of parameters:
void display(int a, double b) { ... }
void display(double a, int b) { ... }

Example:
class Calculator {
    // Method overloading
    int add(int a, int b) {
        return a + b;
    }

    double add(double a, double b) {
        return a + b;
    }

    int add(int a, int b, int c) {
        return a + b + c;
    }
}

Benefits:
• Improves code readability
• Provides flexibility to users
• Reduces method name complexity
• Achieved at compile time (static polymorphism)
The main method is the entry point of any Java application. It's the first method that gets executed when you run a Java program.

Signature:
public static void main(String[] args) {
    // Program execution starts here
}

Components Breakdown:
• public: Accessible from anywhere
• static: Can be called without creating an object
• void: Doesn't return any value
• main: Method name (fixed)
• String[] args: Command-line arguments

Why these modifiers?
• public: JVM needs to access it from outside the class
• static: JVM can call it without instantiating the class
• void: Main method doesn't need to return anything
• String[] args: Allows passing arguments from command line

Example:
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");

        // Access command line arguments
        if (args.length > 0) {
            System.out.println("Arguments: " + args[0]);
        }
    }
}

Running the program:
javac HelloWorld.java
java HelloWorld arg1 arg2
Primitive data types are the basic building blocks of data in Java. They are predefined by the language and are not objects.

• byte: 8-bit signed (-128 to 127)
• short: 16-bit signed (-32,768 to 32,767)
• int: 32-bit signed (-2^31 to 2^31-1)
• long: 64-bit signed (-2^63 to 2^63-1)

Floating-Point Types:
• float: 32-bit IEEE 754 (6-7 decimal digits)
• double: 64-bit IEEE 754 (15 decimal digits)

Other Types:
• char: 16-bit Unicode character
• boolean: true or false

Default Values:
byte b;     // 0
short s;    // 0
int i;      // 0
long l;     // 0L
float f;    // 0.0f
double d;   // 0.0
char c;     // '\u0000'
boolean bool; // false

Key Points:
• Stored in stack memory
• Have default values
• Cannot be null
• Faster than objects
• Used for performance-critical operations
A loop is a programming construct that allows you to execute a block of code repeatedly until a certain condition is met.

Types of Loops in Java:

1. For Loop:
• Used when number of iterations is known
• Combines initialization, condition, and increment
for (int i = 0; i < 5; i++) {
    System.out.println("Count: " + i);
}

2. While Loop:
• Entry-controlled loop
• Condition checked before execution
int i = 0;
while (i < 5) {
    System.out.println("Count: " + i);
    i++;
}

3. Do-While Loop:
• Exit-controlled loop
• Executes at least once
int i = 0;
do {
    System.out.println("Count: " + i);
    i++;
} while (i < 5);

4. Enhanced For Loop (For-Each):
• Used with arrays and collections
• Simplified syntax
int[] numbers = {1, 2, 3, 4, 5};
for (int num : numbers) {
    System.out.println(num);
}

Loop Control Statements:
• break: Exit the loop immediately
• continue: Skip current iteration and continue
• return: Exit the method entirely

Infinite Loops:
// These create infinite loops
while (true) { ... }
for (;;) { ... }
All three classes are used for string manipulation, but they differ in mutability, thread-safety, and performance.

String:
• Immutable: Cannot be changed after creation
• Thread-safe: Safe for multi-threaded environments
• Memory: Creates new object for each modification
• Performance: Slower for frequent modifications
• Use when: String content won't change often

StringBuffer:
• Mutable: Can be modified after creation
• Thread-safe: Synchronized methods
• Memory: Modifies existing object
• Performance: Slower than StringBuilder due to synchronization
• Use when: Multi-threaded string manipulation

StringBuilder:
• Mutable: Can be modified after creation
• Not thread-safe: No synchronization
• Memory: Modifies existing object
• Performance: Fastest for string operations
• Use when: Single-threaded string manipulation

Performance Comparison:
// String - Creates 1001 objects
String str = "";
for(int i = 0; i < 1000; i++) {
    str += i;  // New object each time
}

// StringBuilder - Creates 1 object
StringBuilder sb = new StringBuilder();
for(int i = 0; i < 1000; i++) {
    sb.append(i);  // Modifies same object
}
String result = sb.toString();

Common Methods:
• append() - Add content
• insert() - Insert at position
• delete() - Remove characters
• reverse() - Reverse string
• toString() - Convert to String

Choosing the Right One:
• Use String for constants and single operations
• Use StringBuilder for single-threaded string building
• Use StringBuffer for multi-threaded string building
Object-Oriented Programming (OOP) is based on four fundamental principles that help in creating modular, maintainable, and scalable software. These principles are the foundation of OOP languages like Java.

1. Encapsulation:
• Bundling data and methods that operate on that data within a single unit (class)
• Hiding internal implementation details from outside world
• Providing controlled access through public methods

2. Inheritance:
• Mechanism where one class acquires properties and behaviors of another class
• Promotes code reusability and establishes relationships between classes
• Creates a hierarchy of classes

3. Polymorphism:
• Ability of an object to take many forms
• Same method name can behave differently based on object type
• Achieved through method overloading and overriding

4. Abstraction:
• Hiding complex implementation details and showing only essential features
• Focuses on what an object does rather than how it does it
• Achieved through abstract classes and interfaces

Benefits of OOP Principles:
• Modularity: Code is organized into independent units
• Reusability: Existing code can be reused
• Maintainability: Easier to modify and extend
• Security: Data hiding prevents unauthorized access
• Scalability: Easy to add new features
Class:
• Blueprint/Template: A class is a blueprint or template for creating objects
• Logical Entity: It is a logical entity that defines the structure and behavior
• Declaration: Defined using the class keyword
• Memory: Doesn't occupy memory when declared
• Single Copy: Only one copy exists in memory
• Components: Contains fields (variables) and methods

Object:
• Instance: An object is an instance of a class
• Physical Entity: It is a physical entity that occupies memory
• Creation: Created using the new keyword
• Memory: Occupies memory when created
• Multiple Copies: Multiple objects can be created from same class
• Usage: Used to access class members

Example:
// Class (Blueprint)
class Car {
    String color;
    String model;
    
    void start() {
        System.out.println("Car started");
    }
}

// Objects (Instances)
Car car1 = new Car();  // First object
Car car2 = new Car();  // Second object

car1.color = "Red";
car2.color = "Blue";

Key Points:
• Class is like a recipe, object is like the actual dish
• Multiple objects can be created from one class
• Each object has its own copy of instance variables
Encapsulation is the process of wrapping data (variables) and methods that operate on that data into a single unit called a class. It hides the internal implementation details from the outside world and provides controlled access through public methods.

Key Concepts:
• Data Hiding: Internal data is hidden from external access
• Access Control: Controlled access through getter/setter methods
• Security: Prevents unauthorized access to data
• Modularity: Changes to internal implementation don't affect external code

Example - Bank Account:
class BankAccount {
    // Private variables (encapsulated data)
    private String accountNumber;
    private double balance;
    private String accountHolder;
    
    // Constructor
    public BankAccount(String accNum, String holder) {
        this.accountNumber = accNum;
        this.accountHolder = holder;
        this.balance = 0.0;
    }
    
    // Public getter methods (controlled access)
    public String getAccountNumber() {
        return accountNumber;
    }
    
    public double getBalance() {
        return balance;
    }
    
    public String getAccountHolder() {
        return accountHolder;
    }
    
    // Public setter methods with validation
    public void setAccountHolder(String holder) {
        if (holder != null && !holder.trim().isEmpty()) {
            this.accountHolder = holder;
        }
    }
    
    // Public methods for operations
    public void deposit(double amount) {
        if (amount > 0) {
            balance += amount;
            System.out.println("Deposited: $" + amount);
        }
    }
    
    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
            System.out.println("Withdrawn: $" + amount);
        } else {
            System.out.println("Invalid withdrawal amount");
        }
    }
}

Usage:
BankAccount account = new BankAccount("123456", "John Doe");
account.deposit(1000);
account.withdraw(500);
System.out.println("Balance: $" + account.getBalance());

Benefits:
• Protects data integrity
• Allows validation of data
• Provides flexibility to change implementation
• Improves maintainability
Inheritance is a mechanism in OOP where one class (child/subclass) acquires the properties and behaviors of another class (parent/superclass). It promotes code reusability and establishes a relationship between classes.

Basic Syntax:
class ChildClass extends ParentClass {
    // Child class can use parent's methods and variables
    // Can also add new methods or override existing ones
}

Types of Inheritance:

1. Single Inheritance:
• One child class inherits from one parent class
class Animal { }
class Dog extends Animal { }

2. Multilevel Inheritance:
• Child class becomes parent for another class
class Animal { }
class Dog extends Animal { }
class Puppy extends Dog { }

3. Hierarchical Inheritance:
• Multiple child classes inherit from one parent class
class Animal { }
class Dog extends Animal { }
class Cat extends Animal { }
class Cow extends Animal { }

4. Multiple Inheritance (Not supported in Java with classes):
• One child class inherits from multiple parent classes
• Supported through interfaces in Java

Example:
class Vehicle {
    String brand;
    int speed;
    
    void start() {
        System.out.println("Vehicle started");
    }
}

class Car extends Vehicle {
    int wheels = 4;
    
    void drive() {
        System.out.println("Car is driving");
    }
}

class ElectricCar extends Car {
    int batteryCapacity;
    
    void charge() {
        System.out.println("Charging electric car");
    }
}

Key Keywords:
• extends - Used to inherit from a class
• super - Refers to parent class members
• final - Prevents inheritance

Benefits:
• Code reusability
• Method overriding
• Establishes IS-A relationship
• Reduces code duplication
Method Overloading:
• Compile-time Polymorphism: Resolved during compilation
• Same Class: Methods are in the same class
• Method Signature: Same name, different parameters
• Inheritance: Not required
• Return Type: Can be different
• Access Modifier: Can be different
• Purpose: Increase readability and reusability

Method Overriding:
• Runtime Polymorphism: Resolved during runtime
• Different Classes: Methods are in different classes (inheritance required)
• Method Signature: Must be exactly same
• Inheritance: Required (extends or implements)
• Return Type: Must be same or covariant
• Access Modifier: Must be same or more accessible
• Purpose: Provide specific implementation

Overloading Example Overriding Example 
class Calculator {
    // Same method name, different parameters
    int add(int a, int b) {
        return a + b;
    }
    
    double add(double a, double b) {
        return a + b;
    }
    
    int add(int a, int b, int c) {
        return a + b + c;
    }
}
 
class Animal {
    void makeSound() {
        System.out.println("Animal makes sound");
    }
}

class Dog extends Animal {
    // Same signature, different implementation
    @Override
    void makeSound() {
        System.out.println("Dog barks");
    }
}
Key Differences:
• Overloading: Same class, different parameters (highlighted in yellow)
• Overriding: Different classes, same signature (highlighted in yellow)
• Overloading: Compile-time binding
• Overriding: Runtime binding
Abstract Class:
• Partial Abstraction: Can have both abstract and concrete methods
• Variables: Can have instance variables, static variables, constants
• Constructors: Can have constructors
• Access Modifiers: Can use all access modifiers
• Inheritance: Supports single inheritance
• Keyword: abstract
• Usage: When classes share common behavior

Interface:
• Complete Abstraction: All methods are abstract (before Java 8)
• Variables: Only public static final constants
• Constructors: Cannot have constructors
• Access Modifiers: All members are public by default
• Inheritance: Supports multiple inheritance
• Keyword: interface
• Usage: When classes have different implementations

Abstract Class Example Interface Example 
abstract class Shape {
    protected String color;
    
    // Constructor
    public Shape(String color) {
        this.color = color;
    }
    
    // Abstract method
    abstract double calculateArea();
    
    // Concrete method
    public void displayColor() {
        System.out.println("Color: " + color);
    }
}

class Circle extends Shape {
    private double radius;
    
    public Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }
    
    @Override
    double calculateArea() {
        return Math.PI * radius * radius;
    }
}
 
interface Drawable {
    // Constant
    String TOOL = "PEN";
    
    // Abstract methods
    void draw();
    void erase();
    
    // Default method (Java 8+)
    default void resize() {
        System.out.println("Resizing shape");
    }
    
    // Static method (Java 8+)
    static void printTool() {
        System.out.println("Using: " + TOOL);
    }
}

class Rectangle implements Drawable {
    @Override
    public void draw() {
        System.out.println("Drawing rectangle");
    }
    
    @Override
    public void erase() {
        System.out.println("Erasing rectangle");
    }
}
When to Use:
• Abstract Class: When you want to share code among closely related classes
• Interface: When you want to define a contract for unrelated classes
Runtime polymorphism, also known as dynamic polymorphism or late binding, is a process where a call to an overridden method is resolved at runtime rather than compile time. It occurs when a superclass reference refers to a subclass object.

Key Characteristics:
• Method Overriding: Achieved through method overriding
• Inheritance: Required between classes
• Runtime Resolution: Method call resolved at runtime
• Dynamic Binding: JVM determines which method to call
• Superclass Reference: Reference variable of parent class

Example:
class Animal {
    void makeSound() {
        System.out.println("Animal makes a sound");
    }
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("Dog barks");
    }
}

class Cat extends Animal {
    @Override
    void makeSound() {
        System.out.println("Cat meows");
    }
}

public class Test {
    public static void main(String[] args) {
        // Runtime polymorphism
        Animal animal1 = new Dog();  // Parent reference, child object
        Animal animal2 = new Cat();  // Parent reference, child object
        
        animal1.makeSound();  // Calls Dog's makeSound() - resolved at runtime
        animal2.makeSound();  // Calls Cat's makeSound() - resolved at runtime
    }
}

How it Works:
1. Compile Time: Compiler checks if method exists in parent class
2. Runtime: JVM looks for actual object type and calls appropriate method
3. Dynamic Method Dispatch: JVM decides which method to execute

Rules for Runtime Polymorphism:
• Must have inheritance relationship
• Method must be overridden in child class
• Method signatures must be same
• Return types must be same or covariant
• Access modifiers can be same or more accessible

Benefits:
• Flexibility: Same code works with different objects
• Extensibility: Easy to add new subclasses
• Loose Coupling: Code depends on abstractions
• Code Reusability: Common interface for different implementations

Real-world Example:
// Payment processing system
PaymentProcessor processor = new CreditCardPayment();  // Runtime polymorphism
processor.processPayment(100.0);  // Calls credit card specific method

processor = new PayPalPayment();  // Same reference, different object
processor.processPayment(100.0);  // Calls PayPal specific method
Data hiding is a fundamental concept in OOP that refers to the practice of hiding the internal data of a class from external access and providing controlled access through public methods. It ensures that the data is not directly accessible from outside the class.

Key Concepts:
• Access Control: Restricting direct access to data members
• Encapsulation: Data hiding is achieved through encapsulation
• Security: Protects data from unauthorized access
• Validation: Allows data validation before setting values
• Flexibility: Internal implementation can be changed without affecting external code

Access Modifiers in Java:
• private: Accessible only within the same class
• default (no modifier): Accessible within the same package
• protected: Accessible within package and subclasses
• public: Accessible from anywhere

Example - Without Data Hiding:
class Student {
    public String name;  // Direct access - not recommended
    public int age;
    
    public void display() {
        System.out.println(name + " is " + age + " years old");
    }
}

// Usage - Direct manipulation possible
Student s = new Student();
s.name = "John";     // Direct access
s.age = -5;          // Invalid data allowed
s.display();

Example - With Data Hiding:
class Student {
    private String name;  // Hidden data
    private int age;
    
    // Getter methods
    public String getName() {
        return name;
    }
    
    public int getAge() {
        return age;
    }
    
    // Setter methods with validation
    public void setName(String name) {
        if (name != null && !name.trim().isEmpty()) {
            this.name = name;
        }
    }
    
    public void setAge(int age) {
        if (age > 0 && age < 120) {
            this.age = age;
        }
    }
    
    public void display() {
        System.out.println(name + " is " + age + " years old");
    }
}

// Usage - Controlled access
Student s = new Student();
s.setName("John");   // Validated setter
s.setAge(25);        // Validated setter
s.setAge(-5);        // Invalid - ignored
s.display();

Benefits:
• Data Integrity: Prevents invalid data
• Security: Protects sensitive data
• Maintenance: Easy to modify internal implementation
• Abstraction: Hides complexity from users
• Validation: Ensures data consistency

Best Practices:
• Make instance variables private
• Provide public getter/setter methods
• Validate data in setter methods
• Use appropriate access modifiers
• Document the purpose of each method
Java does not support multiple inheritance with classes (where one class inherits from multiple classes) to avoid the "Diamond Problem" and maintain simplicity in the language design. However, Java supports multiple inheritance through interfaces.

The Diamond Problem:
• Occurs when a class inherits from two classes that have a common method
• Compiler cannot determine which method to call
• Leads to ambiguity and complexity

Example of Diamond Problem:
// This is NOT allowed in Java
class A {
    void display() {
        System.out.println("A");
    }
}

class B {
    void display() {
        System.out.println("B");
    }
}

// This would cause ambiguity
class C extends A, B {  // Not allowed
    // Which display() method should C inherit?
    // From A or from B?
}

Why Java Avoided It:
• Complexity: Makes language complex
• Ambiguity: Diamond problem causes confusion
• Performance: Slower compilation and execution
• Maintainability: Harder to maintain and debug
• Simplicity: Java focuses on simplicity

Java's Solution - Interfaces:
• Supports multiple inheritance through interfaces
• No implementation conflicts
• Only method signatures, no method bodies (before Java 8)
• Diamond problem solved through explicit implementation

Multiple Inheritance with Interfaces:
interface Printable {
    void print();
}

interface Showable {
    void show();
}

class Document implements Printable, Showable {
    @Override
    public void print() {
        System.out.println("Printing document");
    }
    
    @Override
    public void show() {
        System.out.println("Showing document");
    }
}

Why Interfaces Work:
• No method implementations in interfaces (mostly)
• Class provides all implementations
• No ambiguity about which method to call
• Each method is explicitly implemented

Alternative Solutions in Java:
• Composition: Use has-a relationship instead of is-a
• Interfaces: Multiple interface implementation
• Inner Classes: For complex inheritance scenarios
• Delegation: Forward method calls to other objects

Real-world Analogy:
• Single Inheritance: Like a child inheriting traits from one parent
• Multiple Inheritance: Like a child inheriting traits from two parents (can be confusing)
• Interfaces: Like learning skills from multiple teachers (clear and specific)
An interface in Java is a reference type that is similar to a class but contains only abstract methods, default methods, static methods, and constants. It defines a contract that implementing classes must follow, providing complete abstraction.

Key Characteristics:
• Abstract Methods: Methods without implementation
• Constants: Public static final variables
• Default Methods: Methods with default implementation (Java 8+)
• Static Methods: Class-level methods (Java 8+)
• Private Methods: Helper methods (Java 9+)
• Multiple Inheritance: A class can implement multiple interfaces

Basic Syntax:
interface InterfaceName {
    // Constants (public static final by default)
    int MAX_VALUE = 100;
    
    // Abstract methods (public abstract by default)
    void method1();
    void method2();
    
    // Default method (Java 8+)
    default void defaultMethod() {
        System.out.println("Default implementation");
    }
    
    // Static method (Java 8+)
    static void staticMethod() {
        System.out.println("Static method");
    }
}

Implementation:
class MyClass implements InterfaceName {
    @Override
    public void method1() {
        System.out.println("Implementation of method1");
    }
    
    @Override
    public void method2() {
        System.out.println("Implementation of method2");
    }
    
    // Can override default method if needed
    @Override
    public void defaultMethod() {
        System.out.println("Overridden default method");
    }
}

Types of Interfaces:

1. Normal Interface:
• Contains abstract methods and constants
• Implementing class must provide all implementations

2. Functional Interface (Java 8+):
• Contains exactly one abstract method
• Can be used with lambda expressions
• Marked with @FunctionalInterface annotation
@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);
}

// Usage with lambda
Calculator add = (a, b) -> a + b;
Calculator multiply = (a, b) -> a * b;

3. Marker Interface:
• Empty interface with no methods
• Used to mark classes for special treatment
• Example: Serializable, Cloneable

Interface vs Abstract Class:
• Multiple Inheritance: Interface supports, abstract class doesn't
• Variables: Interface has only constants, abstract class can have instance variables
• Constructors: Interface can't have, abstract class can
• Method Types: Interface has abstract, default, static; abstract class has all types

Benefits:
• Abstraction: Hides implementation details
• Multiple Inheritance: Solves diamond problem
• Loose Coupling: Reduces dependencies
• Testability: Easy to mock for testing
• Flexibility: Allows different implementations

Best Practices:
• Use descriptive names ending with "able" (Runnable, Serializable)
• Keep interfaces focused on single responsibility
• Use default methods for backward compatibility
• Document abstract methods with JavaDoc
• Consider functional interfaces for lambda support
Abstraction is one of the fundamental principles of Object-Oriented Programming (OOP) that focuses on hiding complex implementation details and showing only the essential features of an object. It allows programmers to focus on what an object does rather than how it does it.

Key Concepts of Abstraction:
• Data Abstraction: Hiding data implementation details
• Method Abstraction: Hiding method implementation details
• Interface: Contract that defines what methods should be implemented
• Abstract Class: Partially implemented class that cannot be instantiated

Types of Abstraction:

1. Data Abstraction:
• Hides internal data representation
• Provides controlled access through methods
• Achieved using access modifiers (private, protected)

2. Method Abstraction:
• Hides method implementation details
• Shows only method signatures
• Achieved using abstract methods and interfaces

Polymorphism is a core concept in Object-Oriented Programming that allows objects of different classes to be treated as objects of a common superclass. The word "polymorphism" comes from Greek, meaning "many forms." It enables a single interface to represent different underlying forms (data types).

Key Concepts:
• Many Forms: Same method name, different implementations
• Dynamic Binding: Method call resolved at runtime
• Static Binding: Method call resolved at compile time
• Method Overriding: Runtime polymorphism
• Method Overloading: Compile-time polymorphism

Types of Polymorphism:

1. Compile-time Polymorphism (Static Binding):
• Also called static polymorphism or early binding
• Method overloading is the most common example
• Resolved during compilation
• Faster execution but less flexible

2. Runtime Polymorphism (Dynamic Binding):
• Also called dynamic polymorphism or late binding
• Method overriding is the most common example
• Resolved during execution
• More flexible but slightly slower

Runtime Polymorphism Example:
public class AnimalDemo {
    public static void main(String[] args) {
        // Array of Animal references - polymorphism in action!
        Animal[] animals = new Animal[4];
        
        animals[0] = new Dog("Buddy", 3, "Golden Retriever");
        animals[1] = new Cat("Whiskers", 2, "Black");
        animals[2] = new Bird("Tweety", 1, true);
        animals[3] = new Dog("Max", 5, "German Shepherd");
        
        // Polymorphic behavior - same method call, different implementations
        System.out.println("=== Animal Sounds (Polymorphism) ===");
        for (Animal animal : animals) {
            animal.makeSound();  // Which makeSound() is called? Determined at runtime!
        }
        
        System.out.println("\n=== Common Behaviors ===");
        for (Animal animal : animals) {
            animal.eat();    // Same implementation for all
            animal.sleep();  // Same implementation for all
        }
        
        System.out.println("\n=== Specific Behaviors ===");
        // Downcasting to access specific methods
        for (Animal animal : animals) {
            if (animal instanceof Dog) {
                ((Dog) animal).fetch();
            } else if (animal instanceof Cat) {
                ((Cat) animal).climb();
            } else if (animal instanceof Bird) {
                ((Bird) animal).fly();
            }
        }
    }
}

Inheritance is a fundamental concepts in Object-Oriented Programming where a new class (child or subclass) is created from an existing class (parent or superclass). The child class inherits all the properties and behaviors of the parent class and can add new features or modify existing ones. It establishes an "IS-A" relationship between classes.

Key Concepts:
• Parent/Superclass: The class being inherited from
• Child/Subclass: The class that inherits
• IS-A Relationship: A Dog IS-A Animal, A Car IS-A Vehicle
• Code Reusability: Avoid duplicating code
• Method Overriding: Changing parent method behavior
• Access Modifiers: Control inheritance visibility

Types of Inheritance:

1. Single Inheritance:
• One child class inherits from one parent class
• Most common and straightforward
• Example: Dog extends Animal

2. Multilevel Inheritance:
• Child class becomes parent for another class
• Creates a chain of inheritance
• Example: Puppy extends Dog extends Animal

3. Hierarchical Inheritance:
• Multiple child classes inherit from one parent class
• Creates a tree-like structure
• Example: Dog, Cat, Cow all extend Animal

4. Multiple Inheritance (Not supported in Java):
• One child class inherits from multiple parent classes
• Supported through interfaces in Java
• Example: FlyingCar extends Car and Airplane (not allowed)

        for (Vehicle vehicle : vehicles) {
            vehicle.start();
            vehicle.stop();
            
            // Specific actions based on type
            if (vehicle instanceof Car) {
                if (vehicle instanceof ElectricCar) {
                    ((ElectricCar) vehicle).charge();
                    ((ElectricCar) vehicle).regenerativeBraking();
                } else {
                    ((Car) vehicle).drive();
                    ((Car) vehicle).honk();
                }
            } else if (vehicle instanceof Motorcycle) {
                ((Motorcycle) vehicle).wheelie();
                ((Motorcycle) vehicle).race();
            }
            System.out.println();
        }
    }
}

Benefits of Inheritance:
• Code Reusability: Avoid rewriting common code
• Extensibility: Easy to add new features
• Maintainability: Changes in parent class affect all children
• Polymorphism: Enables runtime polymorphism
• Hierarchy: Natural representation of real-world relationships
• Modularity: Organize code in logical groups

Access Modifiers in Inheritance:
• public: Accessible everywhere
• protected: Accessible in same package and subclasses
• default: Accessible in same package only
• private: Accessible only in same class

Key Keywords:
• extends: Used to inherit from a class
• super: Refers to parent class constructor/methods
• final: Prevents inheritance or method overriding
• abstract: Cannot be instantiated, must be extended

Real-World Examples:
• Animal Hierarchy: Animal → Mammal → Dog
• Vehicle Types: Vehicle → Car → SportsCar
• Employee Types: Employee → Manager, Developer, Intern
• Shape Hierarchy: Shape → Circle, Rectangle, Triangle

Best Practices:
• Use inheritance only when IS-A relationship exists
• Prefer composition over inheritance when HAS-A relationship exists
• Keep inheritance hierarchies shallow (not too deep)
• Use abstract classes for partial implementation
• Use interfaces for multiple inheritance
• Override methods appropriately with @Override annotation
• Document inheritance relationships clearly
Encapsulation is a fundamental principle of Object-Oriented Programming that involves bundling data (attributes) and methods (functions) that operate on that data into a single unit called a class. It restricts direct access to some of an object's components and provides controlled access through public methods. The main goal is to protect the internal state of an object from unauthorized access and modification.

Key Concepts:
• Data Hiding: Internal data is hidden from external access
• Access Control: Controlled access through getter/setter methods
• Information Hiding: Hide implementation details
• Modularity: Self-contained units with clear interfaces
• Security: Prevent unauthorized data manipulation

Access Modifiers in Java:
• private: Accessible only within the same class
• default (no modifier): Accessible within the same package
• protected: Accessible within package and subclasses
• public: Accessible from anywhere

Detailed Example - Student Management System:
class Student {
    // Private instance variables (encapsulated data)
    private String studentId;
    private String name;
    private int age;
    private String grade;
    private double gpa;
    private List<String> enrolledCourses;
    
    // Constructor
    public Student(String studentId, String name, int age) {
        this.studentId = studentId;
        this.name = name;
        this.age = age;
        this.grade = "Not Assigned";
        this.gpa = 0.0;
        this.enrolledCourses = new ArrayList<>();
    }
    
    // Public getter methods (controlled read access)
    public String getStudentId() {
        return studentId;
    }
    
    public String getName() {
        return name;
    }
    
    public int getAge() {
        return age;
    }
    
    public String getGrade() {
        return grade;
    }
    
    public double getGpa() {
        return gpa;
    }
    
    public List<String> getEnrolledCourses() {
        // Return a copy to prevent external modification
        return new ArrayList<>(enrolledCourses);
    }
    
    // Public setter methods with validation (controlled write access)
    public void setName(String name) {
        if (name != null && !name.trim().isEmpty() && name.length() >= 2) {
            this.name = name.trim();
        } else {
            throw new IllegalArgumentException("Name must be at least 2 characters long");
        }
    }
    
    public void setAge(int age) {
        if (age >= 5 && age <= 100) {
            this.age = age;
        } else {
            throw new IllegalArgumentException("Age must be between 5 and 100");
        }
    }
    
    public void setGrade(String grade) {
        if (grade != null && (grade.equals("A") || grade.equals("B") || grade.equals("C") || 
            grade.equals("D") || grade.equals("F"))) {
            this.grade = grade;
        } else {
            throw new IllegalArgumentException("Grade must be A, B, C, D, or F");
        }
    }
    
    // Public methods for operations (business logic)
    public void enrollCourse(String courseName) {
        if (courseName != null && !courseName.trim().isEmpty()) {
            if (!enrolledCourses.contains(courseName)) {
                enrolledCourses.add(courseName);
                System.out.println(name + " enrolled in " + courseName);
            } else {
                System.out.println(name + " is already enrolled in " + courseName);
            }
        }
    }
    
    public void dropCourse(String courseName) {
        if (enrolledCourses.remove(courseName)) {
            System.out.println(name + " dropped " + courseName);
        } else {
            System.out.println(name + " is not enrolled in " + courseName);
        }
    }
    
    public void updateGPA(double newGPA) {
        if (newGPA >= 0.0 && newGPA <= 4.0) {
            this.gpa = newGPA;
            // Automatically update grade based on GPA
            updateGradeFromGPA();
        } else {
            throw new IllegalArgumentException("GPA must be between 0.0 and 4.0");
        }
    }
    
    // Private helper method (completely hidden from outside)
    private void updateGradeFromGPA() {
        if (gpa >= 3.7) grade = "A";
        else if (gpa >= 3.0) grade = "B";
        else if (gpa >= 2.0) grade = "C";
        else if (gpa >= 1.0) grade = "D";
        else grade = "F";
    }
    
    // Public method to display complete information
    public void displayStudentInfo() {
        System.out.println("=== Student Information ===");
        System.out.println("ID: " + studentId);
        System.out.println("Name: " + name);
        System.out.println("Age: " + age);
        System.out.println("Grade: " + grade);
        System.out.println("GPA: " + gpa);
        System.out.println("Enrolled Courses: " + enrolledCourses);
        System.out.println("===========================");
    }
}

Advanced Example - Bank Account with Security:
class SecureBankAccount {
    // Encapsulated sensitive data
    private String accountNumber;
    private String accountHolder;
    private double balance;
    private String pin; // Encrypted
    private boolean isLocked;
    private List<String> transactionHistory;
    private int failedAttempts;
    
    // Constants for security
    private static final int MAX_FAILED_ATTEMPTS = 3;
    private static final double MINIMUM_BALANCE = 100.0;
    
    public SecureBankAccount(String accountNumber, String accountHolder, String pin) {
        this.accountNumber = accountNumber;
        this.accountHolder = accountHolder;
        this.balance = 0.0;
        this.pin = encryptPin(pin); // Encrypt PIN for security
        this.isLocked = false;
        this.transactionHistory = new ArrayList<>();
        this.failedAttempts = 0;
    }
    
    // Public interface methods
    public boolean deposit(double amount, String enteredPin) {
        if (!authenticate(enteredPin)) {
            return false;
        }
        
        if (amount > 0) {
            balance += amount;
            recordTransaction("Deposit: $" + amount);
            resetFailedAttempts();
            return true;
        }
        return false;
    }
    
    public boolean withdraw(double amount, String enteredPin) {
        if (!authenticate(enteredPin)) {
            return false;
        }
        
        if (amount > 0 && (balance - amount) >= MINIMUM_BALANCE) {
            balance -= amount;
            recordTransaction("Withdrawal: $" + amount);
            resetFailedAttempts();
            return true;
        }
        return false;
    }
    
    public double getBalance(String enteredPin) {
        if (authenticate(enteredPin)) {
            resetFailedAttempts();
            return balance;
        }
        return -1; // Error indicator
    }
    
    public boolean transfer(double amount, String recipient, String enteredPin) {
        if (!authenticate(enteredPin)) {
            return false;
        }
        
        if (amount > 0 && (balance - amount) >= MINIMUM_BALANCE) {
            balance -= amount;
            recordTransaction("Transfer to " + recipient + ": $" + amount);
            resetFailedAttempts();
            return true;
        }
        return false;
    }
    
    public List<String> getTransactionHistory(String enteredPin) {
        if (authenticate(enteredPin)) {
            resetFailedAttempts();
            return new ArrayList<>(transactionHistory); // Return copy
        }
        return new ArrayList<>(); // Empty list on failure
    }
    
    // Private security methods (completely hidden)
    private boolean authenticate(String enteredPin) {
        if (isLocked) {
            System.out.println("Account is locked due to too many failed attempts");
            return false;
        }
        
        if (enteredPin.equals(pin)) {
            resetFailedAttempts();
            return true;
        } else {
            failedAttempts++;
            if (failedAttempts >= MAX_FAILED_ATTEMPTS) {
                lockAccount();
            }
            System.out.println("Invalid PIN. Attempts remaining: " + (MAX_FAILED_ATTEMPTS - failedAttempts));
            return false;
        }
    }
    
    private void lockAccount() {
        isLocked = true;
        System.out.println("Account locked for security reasons");
    }
    
    private void resetFailedAttempts() {
        failedAttempts = 0;
        isLocked = false;
    }
    
    private void recordTransaction(String transaction) {
        String timestamp = java.time.LocalDateTime.now().toString();
        transactionHistory.add(timestamp + " - " + transaction);
    }
    
    private String encryptPin(String pin) {
        // Simple encryption for demonstration (use proper encryption in real apps)
        return "ENC_" + pin.hashCode();
    }
    
    // Public method to change PIN
    public boolean changePin(String oldPin, String newPin) {
        if (authenticate(oldPin)) {
            if (newPin.length() == 4 && newPin.matches("\\d{4}")) {
                pin = encryptPin(newPin);
                recordTransaction("PIN changed");
                resetFailedAttempts();
                return true;
            }
        }
        return false;
    }
}

• Data Security: Prevents unauthorized access to sensitive data
• Data Integrity: Validation ensures data consistency
• Flexibility: Internal implementation can change without affecting users
• Maintainability: Easier to modify and debug code
• Modularity: Self-contained units with clear boundaries

Common Encapsulation Patterns:

1. JavaBeans Pattern:
class Person {
    private String name;
    private int age;
    
    // Getters and setters
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public int getAge() { return age; }
    public void setAge(int age) { this.age = age; }
}

2. Immutable Objects:
class ImmutablePerson {
    private final String name;
    private final int age;
    
    public ImmutablePerson(String name, int age) {
        this.name = name;
        this.age = age;
    }
    
    // Only getters, no setters
    public String getName() { return name; }
    public int getAge() { return age; }
}

3. Builder Pattern for Complex Objects:
class Computer {
    private String cpu;
    private int ram;
    private int storage;
    private String gpu;
    
    private Computer(Builder builder) {
        this.cpu = builder.cpu;
        this.ram = builder.ram;
        this.storage = builder.storage;
        this.gpu = builder.gpu;
    }
    
    // Getters only
    public String getCpu() { return cpu; }
    // ... other getters
    
    public static class Builder {
        private String cpu;
        private int ram;
        private int storage;
        private String gpu;
        
        public Builder cpu(String cpu) { this.cpu = cpu; return this; }
        public Builder ram(int ram) { this.ram = ram; return this; }
        public Builder storage(int storage) { this.storage = storage; return this; }
        public Builder gpu(String gpu) { this.gpu = gpu; return this; }
        
        public Computer build() { return new Computer(this); }
    }
}
